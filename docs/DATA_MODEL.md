# Gamified Habit Tracker — Data Model

**Current status:** Phase 5 complete
**Next phase:** Phase 6 — Dashboard and Streaks
**Database:** PostgreSQL
**ORM:** Entity Framework Core
**Naming:** C# uses PascalCase; PostgreSQL uses `snake_case`

---

## 1. Purpose

This document describes the implemented data model for the Gamified Habit Tracker and the limited post-MVP entity names reserved for later work.

The data model supports the current product loop:

1. a user creates a habit,
2. the user completes that habit for their local date,
3. the backend records the completion,
4. the completion creates XP transactions,
5. the user’s affected attributes gain XP,
6. undoing today’s completion reverses those applied rewards.

The data model must continue to support the approved dashboard and UI vision without adding speculative tables or fields before their behavior exists.

---

## 2. Data-Model Principles

### 2.1 PostgreSQL is the source of truth

Persistent product state belongs in PostgreSQL.

React may hold temporary display state, but it does not permanently own:

- authentication identity,
- habit ownership,
- completion state,
- XP balances,
- reward mappings,
- attribute levels,
- streak state,
- dashboard summaries.

### 2.2 Backend-owned identifiers

Implemented entities use application-generated Guid version 7 identifiers.

Entity Framework Core configures those identifiers with `ValueGeneratedNever()` so PostgreSQL preserves the values generated by the backend.

### 2.3 Historical data is preserved

Habits are soft-deactivated rather than hard-deleted through normal product behavior.

This preserves:

- completion history,
- XP history,
- attribute growth,
- future streak calculations,
- future dashboard summaries.

### 2.4 Calculated values are not duplicated unnecessarily

The database stores authoritative inputs and synchronized aggregates.

It does not currently store separate columns for:

- attribute level,
- overall user level,
- XP inside the current level,
- XP needed for the next level,
- current streak,
- longest streak,
- completion percentage.

Those values are calculated by backend services and returned through DTOs.

### 2.5 Future-compatible, not future-built

The current schema should support the approved MVP screens without adding placeholder entities for:

- avatars,
- currencies,
- quests,
- notifications,
- leaderboards,
- journals,
- AI recommendations,
- public profiles,
- advanced analytics.

## 6. Entity: User

Represents an application account and ownership boundary.

### Fields

- `Id`
- `Email`
- `NormalizedEmail`
- `Username`
- `NormalizedUsername`
- `PasswordHash`
- `CreatedAtUtc`
- `LastLoginAtUtc`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `Email` is required and has a maximum length of 254 characters.
- `NormalizedEmail` is required and has a maximum length of 254 characters.
- `Username` is required and contains 3 to 30 characters.
- `NormalizedUsername` is required and has a maximum length of 30 characters.
- `PasswordHash` is required.
- `CreatedAtUtc` is required.
- `LastLoginAtUtc` is nullable until a successful login occurs.

### Normalization

Before storage and lookup:

- email and username are trimmed,
- normalized forms use consistent invariant uppercase casing,
- passwords are never trimmed or normalized.

User-facing forms retain the original accepted `Email` and `Username` values.

### Relationships

- one `User` has exactly one `UserSettings`,
- one `User` has many `Habit` records,
- one `User` has many `HabitCompletion` records,
- one `User` has many `UserAttribute` records,
- one `User` has many `XpTransaction` records.

Post-MVP:

- one `User` may have many `UserMilestone` records.

### Constraints and indexes

- unique index on `NormalizedEmail`,
- unique index on `NormalizedUsername`,
- required fields reject null values,
- configured maximum lengths match the model,
- identifier is supplied by the application.

### Ownership rule

User-owned API operations derive the current user identifier from authenticated backend claims.

The frontend must not submit an authoritative `UserId` for user-owned operations.

Missing and foreign-owned resources should produce the same non-revealing response where appropriate.

## 7. Entity: UserSettings

Stores user-specific settings required by product behavior.

### Fields

- `Id`
- `UserId`
- `DisplayName`
- `TimeZone`
- `CreatedAtUtc`
- `UpdatedAtUtc`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `UserId` is required.
- `DisplayName` is required and has a maximum length of 50 characters.
- `TimeZone` is required and has a maximum length of 100 characters.
- `CreatedAtUtc` is required.
- `UpdatedAtUtc` is required.

### Relationships

- one `UserSettings` belongs to one `User`,
- each `User` must have exactly one `UserSettings` record.

### Constraints

- unique index on `UserId`,
- required foreign key to `User`,
- deleting the related `User` cascades to `UserSettings`.

### Registration behavior

Registration creates `User` and `UserSettings` in one Entity Framework Core unit of work.

Initial values:

- `DisplayName` equals the accepted username,
- `TimeZone` comes from the validated registration request,
- `CreatedAtUtc` and `UpdatedAtUtc` use the same UTC timestamp.

Both records are saved together or registration fails.

### Time-zone significance

`TimeZone` is domain-significant.

It is used to convert the current UTC timestamp into the user’s local date for:

- habit completion,
- undoing today’s completion,
- future streak calculations,
- future dashboard summaries.

Changing a time zone must not rewrite historical `CompletedDate` values.

## 8. Entity: Habit

Represents a user-owned habit definition.

Examples:

- Go to the gym
- Read a C# chapter
- Meditate
- Avoid nicotine
- Sleep before 11 PM

### Fields

- `Id`
- `UserId`
- `Name`
- `Description`
- `Category`
- `FrequencyType`
- `TargetCount`
- `Difficulty`
- `IsActive`
- `CreatedAtUtc`
- `UpdatedAtUtc`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `UserId` is required.
- `Name` is required, trimmed, and limited to 100 characters.
- `Description` is optional, trimmed, and limited to 500 characters.
- blank description input is stored as `null`.
- `Category` is required and uses `HabitCategory`.
- `FrequencyType` is required and uses `HabitFrequencyType`.
- `TargetCount` is required.
- `Difficulty` is required and uses `HabitDifficulty`.
- `IsActive` is required and defaults to `true`.
- `CreatedAtUtc` and `UpdatedAtUtc` are required UTC timestamps.

### Relationships

- one `Habit` belongs to one `User`,
- one `Habit` has many `HabitCompletion` records,
- one `Habit` has many `HabitAttributeReward` records.

### PostgreSQL mapping

Table:

- `habits`

Important columns:

- `id`
- `user_id`
- `name`
- `description`
- `category`
- `frequency_type`
- `target_count`
- `difficulty`
- `is_active`
- `created_at_utc`
- `updated_at_utc`

### Indexes and constraints

- primary key on `id`,
- index on `(user_id, is_active)`,
- required foreign key to `users(id)`,
- cascade delete from `User`,
- `frequency_type` stored as a string,
- `difficulty` stored as a string,
- `category` stored as a string,
- difficulty check constraint permits `Easy`, `Medium`, `Hard`, or `Elite`,
- frequency/target check constraint enforces:
  - Daily requires `TargetCount = 1`,
  - Weekly requires `TargetCount` from 1 through 7.

The current database configuration does not use free-text categories.

### Soft deactivation

Normal product behavior deactivates a habit by setting `IsActive` to `false`.

It does not delete the row.

This preserves:

- completion history,
- XP transaction history,
- attribute growth,
- future analytics.

### Completion rules

For the MVP:

- Daily habits use `TargetCount = 1`.
- Weekly habits use `TargetCount` from 1 through 7.
- Weekly target count represents intended distinct completion days within a week.
- A habit may be completed at most once per local date.
- Supporting multiple same-day completions would require an explicit future redesign.

## 9. Entity: HabitCompletion

Represents a recorded completion of one habit on one user-local date.

### Fields

- `Id`
- `UserId`
- `HabitId`
- `CompletedDate`
- `CompletedAtUtc`
- `Notes`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `UserId` is required.
- `HabitId` is required.
- `CompletedDate` is required and stored as `DateOnly`.
- `CompletedAtUtc` is required.
- `Notes` is optional, trimmed, and limited to 500 characters.
- blank notes are stored as `null`.
- `UserId` must match the owner of the related habit.

### Relationships

- one `HabitCompletion` belongs to one `User`,
- one `HabitCompletion` belongs to one `Habit`,
- one `HabitCompletion` has many `XpTransaction` records.

### PostgreSQL mapping

Table:

- `habit_completions`

Important columns:

- `id`
- `user_id`
- `habit_id`
- `completed_date`
- `completed_at_utc`
- `notes`

### Indexes and constraints

- unique index on `(habit_id, completed_date)`,
- index on `(user_id, completed_date)`,
- cascade delete from `User`,
- cascade delete from `Habit`,
- identifier supplied by the application.

The unique index is the final protection against concurrent duplicate-completion requests.

### Date distinction

`CompletedDate` is the user-local habit date.

Example:

```text
2026-07-20
```

`CompletedAtUtc` is the exact UTC timestamp when the completion was recorded.

Example:

```text
2026-07-21T01:30:00Z
```

### Creation behavior

The completion request does not accept a client-selected date.

`CompletionService`:

1. loads the owned habit,
2. requires the habit to be active,
3. loads the user’s stored time zone,
4. obtains the current UTC time from `TimeProvider`,
5. calculates the user-local `CompletedDate`,
6. checks for an existing completion,
7. normalizes notes,
8. creates the completion,
9. applies attribute rewards,
10. saves the complete unit of work.

### Undo behavior

Today’s completion may be undone even if the related habit was deactivated after completion.

Undo:

1. confirms the habit belongs to the user,
2. calculates the current local date,
3. loads today’s completion,
4. reverses its XP transactions,
5. removes the XP transactions,
6. removes the completion,
7. saves the unit of work.

## 10. Entity: HabitAttributeReward

Stores the authoritative reward breakdown attached to a habit.

A habit has two current rewards:

- one primary attribute reward,
- one secondary attribute reward.

Example for a Medium Learning and Skills habit:

- Mind: 14 XP
- Focus: 6 XP

### Fields

- `Id`
- `HabitId`
- `AttributeType`
- `XpAmount`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `HabitId` is required.
- `AttributeType` is required.
- `XpAmount` is required and greater than zero.

### Relationships

- one `HabitAttributeReward` belongs to one `Habit`,
- one `Habit` has many `HabitAttributeReward` records.

### PostgreSQL mapping

Table:

- `habit_attribute_rewards`

### Indexes and constraints

- unique index on `(habit_id, attribute_type)`,
- required foreign key to `Habit`,
- cascade delete from `Habit`,
- check constraint requires `xp_amount > 0`,
- check constraint permits only the eight canonical attributes,
- identifier supplied by the application.

### Reward ownership

The client does not submit reward configuration.

`XpService` calculates rewards from:

- `HabitCategory`,
- `HabitDifficulty`.

`HabitService` synchronizes stored reward rows when a habit is created or when category or difficulty changes.

`CompletionService` uses the stored reward rows when awarding completion XP.

A defensive fallback may reconstruct missing legacy reward rows before completion, but the normal stored model contains two reward rows.

### Why rewards are stored

Storing the reward rows supports:

- habit reward preview,
- stable API responses,
- completion reward application,
- future habit-management UI,
- audit-friendly behavior.

The calculation rules remain authoritative in `XpService`.

## 11. Entity: UserAttribute

Stores one user’s current net XP for one canonical character attribute.

### Fields

- `Id`
- `UserId`
- `AttributeType`
- `CurrentXp`
- `UpdatedAtUtc`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `UserId` is required.
- `AttributeType` is required.
- `CurrentXp` is required and cannot be negative.
- `UpdatedAtUtc` is required.

### Relationships

- one `UserAttribute` belongs to one `User`,
- one `User` has many `UserAttribute` records.

### PostgreSQL mapping

Table:

- `user_attributes`

### Indexes and constraints

- unique index on `(user_id, attribute_type)`,
- required foreign key to `User`,
- cascade delete from `User`,
- check constraint requires `current_xp >= 0`,
- check constraint permits only the eight canonical attributes,
- identifier supplied by the application.

### Missing rows

A new user does not need eight physical zero-XP rows immediately.

`AttributeService` returns all eight attributes.

When a physical row is missing, the response uses:

- `CurrentXp = 0`,
- Level 1 progress calculated by the backend.

A row is created when the user first earns XP in that attribute.

### Calculated progression

The database stores `CurrentXp`.

It does not store:

- level,
- XP inside the current level,
- XP needed for the next level.

`XpService` calculates those values.

`AttributeService` returns them through `UserAttributeResponse`.

## 12. Entity: XpTransaction

Represents one currently applied XP award associated with a specific habit completion and attribute.

Example:

- +14 Mind XP from one completion
- +6 Focus XP from the same completion

### Fields

- `Id`
- `UserId`
- `HabitCompletionId`
- `AttributeType`
- `Amount`
- `Reason`
- `CreatedAtUtc`

### Field rules

- `Id` is a backend-generated Guid version 7.
- `UserId` is required.
- `HabitCompletionId` is required.
- `AttributeType` is required.
- `Amount` is required and greater than zero.
- `Reason` is required and limited to 100 characters.
- `CreatedAtUtc` is required.

Current completion transactions use:

```text
Habit completion
```

as the reason.

### Relationships

- one `XpTransaction` belongs to one `User`,
- one `XpTransaction` belongs to one `HabitCompletion`,
- one `HabitCompletion` has many `XpTransaction` records.

### PostgreSQL mapping

Table:

- `xp_transactions`

### Indexes and constraints

- unique index on `(habit_completion_id, attribute_type)`,
- index on `(user_id, created_at_utc)`,
- required foreign key to `User`,
- required foreign key to `HabitCompletion`,
- cascade delete from `User`,
- cascade delete from `HabitCompletion`,
- check constraint requires `amount > 0`,
- check constraint permits only the eight canonical attributes,
- identifier supplied by the application.

### Current transaction semantics

`XpTransaction` represents currently applied XP.

During MVP undo:

- related transaction amounts are subtracted from `UserAttribute.CurrentXp`,
- the related `XpTransaction` rows are removed,
- the related `HabitCompletion` is removed.

The MVP does not keep immutable negative reversal transactions.

Immutable ledger history and explicit reversal records are deferred.

### Why transactions exist

Transactions support:

- auditable reward application,
- exact undo,
- recent XP activity,
- XP gained today,
- future dashboard summaries,
- future grouped analytics,
- debugging data consistency.

## 14. XP and Level Rules

### 14.1 Attribute progression

`UserAttribute.CurrentXp` stores the current net XP for one attribute.

The XP required to advance from the current attribute level is:

```text
100 + 25 × (current level - 1)
```

Examples:

| Current level | XP required for next level |
|---:|---:|
| 1 | 100 |
| 2 | 125 |
| 3 | 150 |
| 4 | 175 |

Attribute level is calculated from total current XP.

It is not stored as a database column.

### 14.2 Overall progression

Overall user level uses a separate progression curve.

The XP required to advance from the current overall level is:

```text
200 + 50 × (current level - 1)
```

Examples:

| Current level | XP required for next level |
|---:|---:|
| 1 | 200 |
| 2 | 250 |
| 3 | 300 |
| 4 | 350 |

Overall level is not stored as a database column.

### 14.3 Overall total XP

The canonical source for currently applied XP awards is `XpTransaction`.

Overall total XP should be derived by the backend from current applied transaction amounts.

`UserAttribute.CurrentXp` is a synchronized aggregate grouped by attribute.

The following invariant should hold:

```text
sum of UserAttribute.CurrentXp
=
sum of current XpTransaction.Amount
```

Phase 6 dashboard tests should protect this assumption when overall progression is exposed.

### 14.4 Frontend rule

The frontend may calculate a visual percentage from backend-returned progress values.

The frontend must not own:

- difficulty totals,
- category mapping,
- reward split,
- level thresholds,
- cumulative level calculations.

## 15. Completion and XP Atomicity

Completion creation and XP application form one persistence unit.

Before the completion save:

- the new `HabitCompletion` is tracked,
- required `UserAttribute` rows are created or updated,
- two `XpTransaction` rows are attached to the completion.

One `SaveChangesAsync` call persists the tracked changes.

If persistence fails, the completion must not intentionally remain without its XP effects.

Undo follows the reverse unit:

- load today’s completion,
- load its XP transactions,
- subtract amounts from user attributes,
- remove XP transactions,
- remove completion,
- call `SaveChangesAsync`.

If reversal validation fails, the unit is not intentionally partially applied.

### Reversal safety checks

Undo fails rather than silently corrupting state when:

- a required user-attribute row is missing,
- a user attribute contains less XP than the related transaction amount.

## 16. Date and Time Rules

Date behavior is critical because the product tracks habits by the user’s local day.

### Current rules

- timestamps are stored in UTC,
- registration requires a valid IANA time-zone identifier,
- the backend validates the identifier,
- `TimeProvider` supplies the current UTC time,
- the backend converts UTC to the user’s local date,
- `CompletedDate` is stored separately from `CompletedAtUtc`,
- completion requests do not accept a client-selected date,
- duplicate completions are prevented by habit and local date,
- historical `CompletedDate` values are not rewritten after time-zone changes.

### Intended usage

Use `CompletedDate` for:

- daily completion state,
- weekly progress,
- streak calculations,
- completion heatmaps,
- date-grouped dashboard summaries.

Use `CompletedAtUtc` for:

- audit ordering,
- recent activity,
- exact completion history,
- debugging.

### Week boundary

The planned MVP week begins on Monday.

Phase 6 streak and weekly-summary implementation must use one documented week-boundary rule consistently in services and tests.

## 17. Source-of-Truth Hierarchy

### Completion

`HabitCompletion` is the source of truth for whether a habit was completed on a date.

`HabitResponse.IsCompletedToday` is a calculated response value.

### Rewards

`XpService` is the source of truth for reward rules.

`HabitAttributeReward` stores the synchronized per-habit reward breakdown.

### Applied XP

`XpTransaction` is the source of truth for currently applied XP awards.

### Attribute totals

`UserAttribute.CurrentXp` is the synchronized per-attribute aggregate.

### Levels

Attribute level and overall level are backend-calculated values.

They are not independent stored truth.

### Streaks

Streaks will be calculated from completion history.

They should not initially be stored as manually trusted counters.

### Dashboard

Dashboard values will be backend aggregates derived from the implemented tables.

React will not create independent persistent dashboard state.

- today’s habits,
- completed and remaining counts,
- completion percentage,
- total current XP,
- overall level,

- overall versus per-habit streaks,
- daily versus weekly habits,
- what counts as a successful day,
- whether an incomplete current day breaks a streak,
- inactive habits,
- week boundaries,
- local time zones,
- historical gaps.

The schema should not be changed until those rules show an actual data requirement.

## 23. Data-Model Change Checklist

Before changing an implemented entity:

1. Explain the behavior that requires the change.
2. Confirm the current phase permits it.
3. Inspect the entity and configuration.
4. Review relationships and delete behavior.
5. Review indexes and constraints.
6. Review DTO impact.
7. Review service logic.
8. Review frontend TypeScript contracts.
9. Add or update migration.
10. Add unit and integration tests.
11. Update this document.
12. Verify existing data compatibility.

Do not:

- delete migrations to hide a model problem,
- drop the development database without understanding the impact,
- rename stable entities casually,
- add nullable placeholder columns for future mockup values,
- store a calculated value merely because the UI displays it.
